## [3.13.2] ŌĆö 2026-03-23

### Critical Fix ŌĆö Sessions Never Tracked (Fatal IPC Bug)
- **Root cause:** `_workerBurstInject` resolves with a raw string (e.g. `'observer-installed'`), but all 3 call sites did `msg.result || 'unknown'` ŌĆö strings have no `.result`, so `result` was always `'unknown'`, causing every session to be silently discarded. `this.sessions` remained permanently empty, the heartbeat loop exited early every tick, and `onClickTelemetry` was never called. Dashboard hardstuck at 7674.
- **Fixed:** All 3 call sites now `await this._workerBurstInject(...) || 'unknown'` directly.
- **Fixed:** Cursor initialized to current `__AA_CLICK_COUNT` (not 0) on session creation to prevent double-counting pre-reload clicks on first heartbeat.

## [3.13.0] ŌĆö 2026-03-23

### Bug Fix ŌĆö Issue #51 (Dashboard "Total Clicks" Counter Stops Incrementing)

Six root causes identified via full code audit + DeepThink architectural review.
Only `ConnectionManager.js` was changed ŌĆö `cdp-worker.js` and `DOMObserver.js` untouched.

- **Fixed (RC1 ŌĆö Critical)** Clicks lost when an Antigravity webview target briefly disappears from `/json`. The heartbeat was pruning sessions from the `sessions` Map *before* harvesting `__AA_CLICK_COUNT`, silently discarding all pending clicks. Sessions going stale are now harvested first with a **1.5 s fast-fail timeout**, then pruned.

- **Fixed (RC2 ŌĆö High)** `ignoredTargets` had no time-based expiry. Targets blocked due to transient injection races (e.g. `no-window` on an unhydrated React DOM) were permanently excluded. Introduced a parallel `_ignoredTargetTTLs` Map ŌĆö non-permanent entries expire after **5 minutes** and are re-tried.

- **Fixed (HB1 ŌĆö Critical)** IPC backpressure death spiral: with 20+ webview targets the heartbeat fired all `_workerEval` calls concurrently, instantly hitting the 20-slot IPC limit. Sessions 21+ were rejected ŌåÆ fail counts ŌåÆ wrongful pruning. All concurrent eval batches are now **chunked at 10**, injections at **5**.

- **Fixed (HB2 ŌĆö Critical)** Destructive read race: `window.__AA_CLICK_COUNT = 0` was written inside the same CDP round-trip that reads the value. If the WebSocket dropped the response, the reset executed but the count was never received ŌåÆ clicks evaporated. Switched to a **monotonic counter + delta cursor** (`_sessionCursors`) ŌĆö the in-page counter is never reset, Node.js tracks a delta.

- **Fixed (HB3 ŌĆö High)** Undead session infinite loop: `_sessionFailCounts.delete(targetId)` was called unconditionally at the top of the `fulfilled` block. Dead observers would accumulate one fail strike, then have it erased on the next heartbeat, trapping them in eternal failed re-injection. Fail count is now cleared **only** when the observer is confirmed alive or re-injection succeeds.

- **Fixed (Secondary)** Removed dead `not-agent-panel` branch in `_handleNewTarget`. `DOMObserver.js` never returns that string ŌĆö the branch was unreachable. Non-agent targets (Launchpad, webview iframe) were always returning `observer-installed` and consuming session slots.

## [3.9.3] ŌĆö 2026-03-12

### Bug Fix ŌĆö Issue #36 (Browser Sub-Agent Crash, TRUE Root Cause)
- **Root cause identified**: The "Cannot freeze array buffer views with elements" crash was NOT caused by WebSocket architecture or CDP session conflicts. It was caused by the MutationObserver **clicking the "Expand" button** during browser sub-agent startup, which opened the `BROWSER_SUBAGENT_TRAJECTORY` modal. The modal's state contains TypedArray screenshot data that AG IDE's state management (`Pin`/`Krt` in `workbench.desktop.main.js`) tried to deep-freeze ŌĆö triggering the crash in the **renderer process** (not extension host).
- **Fixed**: Removed `'expand'` from the auto-click keyword list in `DOMObserver.js`. The "Expand" button is no longer auto-clicked, preventing the trajectory modal from opening prematurely during browser sub-agent initialization.
- **Reverted** all unnecessary architecture changes from v3.8.3ŌĆōv3.9.2 (per-target WebSocket, burst mode, child process isolation, Object.freeze interceptor) since the crash was in the renderer process, not the extension host.

### Bug Fix ŌĆö False Positive "Run" Clicks
- **Fixed**: The MutationObserver was clicking terminal menu items like "Run Task..." and "Run Active File" because the keyword `'run'` with prefix+space matching allowed any text starting with "run " up to 15 chars. Tightened matching for short keywords (< 5 chars) to allow at most `keyword + 4` chars total, preventing menu item false positives while still matching "Run" and "RunAlt+ŌÅÄ".
- **Fixed**: Keyboard shortcut suffix matching now works for short keywords too (lowered threshold from `length >= 5` to `length >= 3`).

## [3.0.0] ŌĆö 2026-02-28

### Architecture: Event-Driven CDP (Zero-Polling)
- **Replaced** attachŌåÆevaluateŌåÆdetach polling cycle with **persistent CDP sessions** (`Map<targetId, sessionId>`)
- **Replaced** periodic script injection with one-shot **MutationObserver** payload ŌĆö reacts instantly when buttons appear in DOM
- **Connection Manager**: browser-level WebSocket stays open, uses `Target.targetCreated`/`Target.targetDestroyed` events for lifecycle
- **Self-healing**: automatic reconnection on WebSocket close, re-injection on execution context clear (webview navigation)
- **Heartbeat**: periodic health check + new target discovery every 30s

### Modularization
- **Split** monolithic 589-line `extension.js` into three modules:
  - `src/extension.js` ŌĆö VS Code lifecycle, command polling, auto-fix patcher
  - `src/cdp/ConnectionManager.js` ŌĆö persistent WebSocket, session pool, target management
  - `src/scripts/DOMObserver.js` ŌĆö MutationObserver payload generator

### Robustness Improvements
- **Deferred Webview Guard**: DOM structure check moved inside `scanAndClick()` to avoid race condition with unhydrated React DOM on `targetCreated`
- **Sequential Polling**: Replaced `setInterval` + async lock with recursive `setTimeout` ŌĆö eliminates lock corruption from overlapping cycles
- **Polling Hang Protection**: `Promise.race` with 3s timeout guarantees the command polling loop can never permanently hang
- **Sibling-Indexed Cooldowns**: `_domPath()` now includes nth-child sibling indices at every DOM level, preventing cooldown collisions when multiple identical buttons appear in a list
- **100ms Throttle**: MutationObserver fires scanAndClick within 100ms of first DOM change (down from 200ms)

### Localized Cooldowns
- **Moved** all cooldown state into the injected DOM script via closure-scoped `clickCooldowns` map
- **Eliminated** Node.js global `lastExpandTimes` map ŌĆö cooldowns are fully per-element

### Continue Button Support (from v2.3.0)
- **Added** automatic clicking of the "Continue" button (agent invocation limit)
